## Dissector Loading System

### Stage 1: Manifest (lib/dissector/manifest.ss)

**What it does**:
- Metadata about all 13 dissectors
- Groups by OSI layer (Link, Network, Transport, Application)
- Associates RFCs, protocol names, and file paths
- Used by build system to determine compilation order

**Data Structure**:
```scheme
'((ethernet "802.3" (link) (ethernet) "dissectors/ethernet.ss")
  (ipv4 "RFC 791" (network) (ipv4) "dissectors/ipv4.ss")
  ;; ... 11 more entries
  (ssh "RFC 4251" (application) (ssh) "dissectors/ssh.ss"))
```

**API**:
```scheme
(dissector-by-name 'dns)              → entry or #f
(dissectors-by-layer 'network)        → filtered list
(dissector-count)                     → 13
(generate-build-list)                 → file paths for compilation
(generate-imports)                    → code for dynamic loading
```

### Stage 2: Loader (lib/dissector/loader.ss)

**What it does**:
- Dynamically imports dissector modules at runtime
- Registers each with protocol registry
- Handles errors gracefully (logs failures, continues)
- Provides `init-dissectors!` entry point

**For Interpreter Mode**:
```scheme
(init-dissectors!)
→ Loads dissectors/*.ss files
→ Registers with protocol registry
→ Returns list of loaded protocols
```

**For Static Binary**:
- Dissectors already linked (pre-compiled)
- `init-dissectors!` still called but mostly a no-op
- All functions available in binary

## Static Binary Build Pipeline

### Step 1: Compile Phase (build-ethereal-musl.ss)

**Input**: Source code + manifest  
**Output**: `.so` and `.wpo` files for each dissector

```scheme
;; Pseudocode from build-ethereal-musl.ss

(for-each (lambda (dissector)
            (compile-dissector dissector 
                              optimize-level: 3
                              wpo: #t))
          (generate-build-list))
```

**Compiles**:
- 13 dissectors: ethernet, ipv4, ipv6, arp, tcp, udp, icmp, icmpv6, igmp, dns, dhcp, ntp, ssh
- 5 libraries: registry, pipeline, flows, statistics, loader
- 1 main tool: wafter (tools/wafter.ss)

### Step 2: Link Phase (musl-gcc)

**Input**: Compiled `.so` files + C entry point  
**Output**: `wafter-musl` static binary

```bash
# Generated by build script
musl-gcc wafter-main.c \
    dissectors/*.so \
    lib/dissector/*.so \
    -static -lm \
    -o wafter-musl
```

**Results**:
- Single ELF binary (~40-60 MB)
- No external dependencies
- Runs on any Linux (glibc/musl/etc)

### Step 3: Verification Phase

**Checks**:
- `file wafter-musl` → "stripped, static"
- `ldd wafter-musl` → "not a dynamic executable" (fails, expected)
- No `/home/` paths in strings
- SHA256 hash for integrity
- Test: `./wafter-musl --version`

## Dissector Registration Flow

### Interpreter Mode (Development)

```
User runs:
  scheme tools/wafter.ss capture.pcap stats

→ tools/wafter.ss imports (jerboa prelude)
  
→ wafter calls (init-dissectors!)
  
→ (init-dissectors!) [from lib/dissector/loader.ss]:
    ├─ Load dissectors/ethernet.ss
    ├─ Load dissectors/ipv4.ss
    ├─ ... (load all 13)
    └─ Each dissector file defines dissect-PROTOCOL function

→ dissect-PROTOCOL functions now in environment
  
→ (register-builtin-dissectors!) [stub in Phase 7]:
    ├─ (register-dissector! 'ethernet dissect-ethernet)
    ├─ (register-dissector! 'ipv4 dissect-ipv4)
    └─ ... (all 13)

→ Protocol registry populated ✓

→ (dissect-packet buffer) uses registry to chain protocols
```

### Static Binary Mode (Phase 7)

```
User runs:
  ./wafter-musl capture.pcap stats

→ Chez runtime boots (embedded in binary)
  
→ Load boot image (pre-compiled modules)
  
→ All 13 dissectors + libraries already linked
  
→ Call (init-dissectors!) [from embedded loader]:
    └─ Mostly a no-op, dissectors already available
    
→ Protocol registry already populated at compile time

→ (dissect-packet buffer) uses registry to chain protocols
```

## Build System Integration

### Dockerfile (Multi-stage)

```dockerfile
# Stage 1: Build
FROM jerboa21/jerboa AS builder
COPY . /build/jerboa-ethereal
WORKDIR /build/jerboa-ethereal
RUN JERBOA_HOME=/opt/jerboa \
    scheme --script build-ethereal-musl.ss

# Stage 2: Runtime (minimal)
FROM ubuntu:24.04
COPY --from=builder /build/jerboa-ethereal/wafter-musl /usr/local/bin/
ENTRYPOINT ["/usr/local/bin/wafter-musl"]
```

### Build Script Execution

```bash
$ make docker                    # Or: make linux-local

→ build-ethereal-musl.ss runs in musl Chez
  
→ Step 1: Check prerequisites
  ✓ musl-gcc found
  ✓ JERBOA_HOME available
  ✓ musl-built Chez found

→ Step 2: Compile dissectors (13 files → 13 .so)
  • manifest.ss provides file list
  • Each compiled with -O3 and WPO
  • Takes ~30-60 seconds

→ Step 3: Generate C entry point
  • Create wafter-main.c
  • Links in boot image
  • Initializes Chez runtime

→ Step 4: Link with musl-gcc
  • Static linking
  • Include math library (-lm)
  • No glibc dependencies

→ Step 5: Strip binary
  • Remove debug symbols
  • Result: ~40-60 MB

→ Step 6: Generate SHA256
  • wafter-musl.sha256 for integrity checking

→ Result: ./wafter-musl ✓ Ready to use
```

## Protocol Chaining Example

Once all dissectors are registered, the pipeline chains them:

```scheme
;; User packet (Ethernet frame containing IPv4 + TCP + DNS)

(dissect-packet #vu8(... ethernet frame ...))

→ Find dissector for 'ethernet
→ (dissect-ethernet buffer)
→ Returns: (protocol . "Ethernet")
          (ethertype . #x0800)
          (payload . ...)

→ Look up ethertype #x0800 → 'ipv4
→ (dissect-ipv4 payload)
→ Returns: (protocol . "IPv4")
          (dst-ip . "8.8.8.8")
          (ip-protocol . 6)
          (payload . ...)

→ Look up ip-protocol 6 → 'tcp
→ (dissect-tcp payload)
→ Returns: (protocol . "TCP")
          (dst-port . 53)
          (payload . ...)

→ Look up dst-port 53 → 'dns
→ (dissect-dns payload)
→ Returns: (protocol . "DNS")
          (query-count . 1)
          (answer-count . 1)
          ...

→ Result: Nested dissection tree with all 4 protocols
```

## Performance Implications

### Interpreter Mode
- Cold start: ~2 seconds (load all dissectors from disk)
- Dissection: 1000+ packets/sec
- Memory: Minimal (only used dissectors loaded)

### Static Binary Mode
- Cold start: <100ms (everything already linked)
- Dissection: 2000+ packets/sec (pre-compiled)
- Memory: Higher startup (~50MB binary), but faster

### Comparison

| Metric | Interpreter | Static Binary |
|--------|-------------|---------------|
| Startup | 2s | <100ms |
| Throughput | 1000 pkt/s | 2000+ pkt/s |
| Binary Size | N/A | 40-60 MB |
| Dependencies | Jerboa + libs | None |
| Use Case | Development | Production |
